- 02 객체지향 프로그래밍을 향해
  - 협력, 객체, 클래스
    - 어떤 클래스가 아니라 어떤 객체가 필요한지 고민하라
     클래스는 공통적인 상태와 행동을 공유하는 객체들을 추상화한 것이다
     어떤 객체가 어떤 상태와 행동을 가질지 먼저 결정하자
    - 객체를 기능을 구현하기 위해 협력하는 공동체의 일원으로 봐야 한다
     객체를 고립된 존재가 아닌 협력에 참여하는 협력자로 보라
     객체들이 모양과 윤곽이 잡히면 공통된 특성과 상태를 가진 객체를 타입으로 분류하고 이 타입을 기반으로 클래스를 구현하라
    - 훌륭한 협력이 훌륭한 객체를 낳고 훌륭한 객체가 훌륭한 클래스를 낳는다
  - 도메인의 구조를 따르는 프로그램 구조
    - 도메인: 문제를 해결하기 위해 사용자가 프로그램을 사용하는 분야
    - 객체지향: 도메인을 구성하는 개념들이 프로그램의 객체와 클래스로 매끄럽게 연결된다
    - 도메인을 구성하는 개념을 관계를 파악(그림2.3)하고 이에 대응하는 클래스와 클래스 관계를 구현(그림2.4)한다
  - 클래스 구현하기
    가시성의 목적: 경계의 명확성이 객체의 자율성과 구현의 자유를 보장함
    - 자율적인 객체
      - 캡슐화: 데이터와 기능을 함께 묶는 것
      - 접근 제어, 접근 수정자: 외부 개입을 최소화
      - 스스로 상태를 관리하고 판단하고 행동하는 자율적인 객체
    - 프로그래머의 자유
      - 클래스 작성자: 데이터 타입을 프로그램에 추가한다
      - 클라이언트 프로그래머: 클래스 작성자가 추가한 타입을 사용한다 
      - 구현 은닉 실현: 변경의 자유로움
  - 협력하는 객체들의 공동체
    - 코드 리뷰
  - 협력에 관한 짧은 이야기
    - 메세지 vs 메서드
    - 메세지를 전송한다 vs 메서드를 호출한다
    - 다형성, 자신만의 방법(구현체)

- 03 할인 요금 구하기
  - 할인 요금 계산을 위한 협력 시작하기
    - Movie 는 DiscountPolicy 에 메세지를 전송할 뿐 할인 정책을 판단하지 않는다
  - 할인 정책과 할인 조건
    - 다형성을 이용하여 여러 할인 정책 구현한다
  - 할인 정책 구성하기
    - 생성자 파라미터로 강제하여 올바른 상태를 가진 객체를 보장한다

- 04 상속과 다형성
  - 컴파일 시간 의존성과 실행 시간 의존성
    - 컴파일 시간 의존성(그림2.7)
    - 실행 시간 의존성(그림2.8, 그림2.9)
    - 두 의존성이 다를수록 코드는 더 유연해지고 확장 가능해진다
  - 차이에 의한 프로그래밍
    - 상속
  - 상속과 인터페이스
    - 다시 한번 생각해볼 메세지 vs 메서드
    - 상속이 가치 있는 이유는 부모 클래스가 제공하는 모든 인터페이스를 자식 클래스가 물려받을 수 있기 때문이다
  - 다형성
    - 동일한 메세지를 수신했을 때 객체의 타입에 따라 다르게 응답할 수 있는 능력을 의미한다
  - 인터페이스와 다형성
    - Interface vs Abstract Class
    - 구현을 공유하는 차이가 있고 업캐스팅, 다형성은 동일하게 적용된다
